Production E-Commerce Blueprint

A complete, opinionated technical reference for building a real e-commerce platform with Django REST + React + MongoDB — from zero to deployed.

Django 4.x REST API React 18 + Redux Toolkit MongoDB + Mongoengine Docker + Nginx + VPS

1 System Architecture

How the Stack Fits Together

Your three main layers are completely decoupled. React talks to Django only via HTTP JSON calls (the REST API). Django talks to MongoDB to persist data. They never touch each other's internals.

User's Browser
React App (SPA)
↕ HTTPS/JSON (JWT in httpOnly Cookie)

↕
Django REST API (Gunicorn)

↕                ↕ (tasks)

Redis (cache + broker)

Each Component Explained

React (Frontend)

A Single Page Application built with Vite + React. It fetches data from Django, renders UI, manages client state. Deployed as static files served by Nginx.

Django (REST API)

Handles all business logic: auth, cart, orders, payments. Uses djangorestframework for endpoints and mongoengine as the MongoDB ODM. Runs via Gunicorn (WSGI server) behind Nginx.

MongoDB

Your database. Flexible documents suit e-commerce well — products have variable attributes, orders embed items, etc. Use Atlas (managed cloud) in production, not a self-hosted instance.

Redis

Two jobs: (1) Cache — store frequently-read data (product lists, category trees) to avoid hitting MongoDB on every request. (2) Message broker — pass tasks to Celery workers.

Celery

Background task runner. Handles emails, order status webhooks, image processing, export jobs — anything that shouldn't block an HTTP response. Runs as a separate process alongside Django.

Nginx

Reverse proxy that sits in front of everything. Serves React's static build files directly, forwards API calls to Gunicorn, handles SSL termination, and serves uploaded media or proxies to S3.

Static & Media Files in Production

  • Static files (React build, CSS, JS): Nginx serves these directly from /dist. Never let Django serve static files in production.
  • User-uploaded media (product images, avatars): Store on Cloudinary or AWS S3. Use django-storages + boto3 to upload directly to S3 from Django. Serve via CloudFront CDN for speed.
  • Why not serve from your server? It's slow, doesn't scale, and wastes your server's bandwidth on binary data.

Deployment: Docker on a VPS

Use Docker Compose on a DigitalOcean Droplet (4GB RAM, $24/month) or equivalent. This is the sweet spot — simpler than Kubernetes, more reproducible than bare metal.

  baseURL: '/api/v1',

// Auto-refresh JWT on 401
api.interceptors.response.use(
  res => res,
  async err => {
    if (err.response?.status === 401 && !err.config._retry) {
      err.config._retry = true;
      await axios.post('/api/v1/auth/token/refresh/', {}, { withCredentials: true });
      return api(err.config);
    }
    return Promise.reject(err);
  }
);

5 Payment Integration

Use Stripe — here's why

Stripe ✓ (Recommended)

  • Better developer experience, excellent docs
  • Stripe Elements keeps card data off your server (PCI compliance handled)
  • Webhook system is reliable and easy to test locally
  • Available in Kenya via Stripe (check current availability)

PayPal / Flutterwave (Alternatives)

  • Flutterwave is excellent for East African markets (M-Pesa, cards)
  • Same webhook pattern applies
  • PayPal is clunkier to integrate but has a huge user base

The Complete Payment Flow

// 1. User clicks "Place Order" on checkout page
// 2. Frontend calls your backend to create a PaymentIntent

## Backend: POST /api/v1/payments/create-intent/
import stripe
stripe.api_key = settings.STRIPE_SECRET_KEY

def create_payment_intent(request):
    order = create_pending_order(request.user, request.data)  # saves order as "pending"
    intent = stripe.PaymentIntent.create(
        amount=int(order.total * 100),   # Stripe uses cents
        currency="usd",
        metadata={ "order_id": str(order.id) },
        idempotency_key=str(order.id)     # prevents duplicate charges on retry
    )
    return Response({ "client_secret": intent.client_secret })

// 3. Frontend uses Stripe.js to collect card and confirm the payment
// The card data goes directly to Stripe servers — never touches your backend
const result = await stripe.confirmCardPayment(clientSecret, {
  payment_method: { card: cardElement }
});

// 4. Stripe sends a webhook to your server when payment succeeds
## Backend: POST /api/v1/payments/webhook/
def stripe_webhook(request):
    payload = request.body
    sig = request.META['HTTP_STRIPE_SIGNATURE']
    # ALWAYS verify the signature — rejects fake webhooks
    event = stripe.Webhook.construct_event(payload, sig, settings.STRIPE_WEBHOOK_SECRET)

    if event['type'] == 'payment_intent.succeeded':
        order_id = event['data']['object']['metadata']['order_id']
        # Mark order as paid, trigger confirmation email via Celery
        update_order_status.delay(order_id, 'paid')
    return HttpResponse(status=200)  # ALWAYS return 200 — or Stripe will retry

Idempotency Keys

Pass idempotency_key=str(order.id) to Stripe. This means if the user double-clicks "Pay" or a network retry fires the request twice, Stripe will return the same PaymentIntent instead of creating a second one. Never charge twice for the same order.

Test Without Real Money

Use Stripe's test card 4242 4242 4242 4242, any future expiry, any CVC. Install the Stripe CLI and run stripe listen --forward-to localhost:8000/api/v1/payments/webhook/ to receive webhooks locally during development.

6 Security & Performance

JWT Storage: httpOnly Cookies (not localStorage)

Never store JWTs in localStorage. JavaScript running on your page (including third-party scripts) can read localStorage — this is an XSS attack vector. httpOnly cookies cannot be read by JavaScript at all. The browser sends them automatically with every request.

Configure in Django: set JWT_AUTH_COOKIE = 'access_token', JWT_AUTH_COOKIE_HTTP_ONLY = True, JWT_AUTH_COOKIE_SAMESITE = 'Lax'. In production, also set JWT_AUTH_COOKIE_SECURE = True (HTTPS only).

Rate Limiting

# Django REST Framework throttling
REST_FRAMEWORK = {
    'DEFAULT_THROTTLE_CLASSES': [
        'rest_framework.throttling.AnonRateThrottle',
        'rest_framework.throttling.UserRateThrottle'
    ],
    'DEFAULT_THROTTLE_RATES': {
        'anon': '100/hour',
        'user': '1000/hour',
        'login': '10/minute',    # tight limit for auth endpoints
    }
}

Input Validation

  • Backend: DRF Serializers validate every incoming field. Never trust frontend data. Validate types, lengths, ranges, and business rules (e.g., quantity > 0, stock available).
  • Frontend: Use Zod schemas with react-hook-form for a good UX — show errors before the user submits. But this is UX only, not security.
  • File uploads: Validate file type (check magic bytes, not just extension) and size server-side before uploading to S3.

Additional Security Checklist

ConcernSolution
CSRF attacksDjango's CSRF middleware + SameSite=Lax cookie. Exempt only the Stripe webhook endpoint (verified by signature instead).
SQL/NoSQL InjectionMongoengine parameterizes queries automatically. Never build raw queries with string concatenation.
Sensitive data exposureNever log full payment data or passwords. Use Django's LOGGING config to filter sensitive fields.
Admin endpoint securityIP whitelist your admin routes in Nginx, or use 2FA.
Dependency vulnerabilitiesRun pip audit and npm audit regularly. GitHub Dependabot handles this automatically.
Secrets managementNever commit .env files. Use environment variables injected at runtime (Docker secrets or platform env vars).

def get_categories():

Email Tasks

# apps/orders/tasks.py
from celery import shared_task
from django.core.mail import send_mail
from django.template.loader import render_to_string

@shared_task(bind=True, max_retries=3)
def send_order_confirmation(self, order_id):
    try:
        order = Order.objects.get(id=order_id)
        html = render_to_string('emails/order_confirmation.html', {'order': order})
        send_mail(
            subject=f"Order {order.order_number} confirmed!",
            message="",
            html_message=html,
            from_email="orders@yourdomain.com",
            recipient_list=[order.user.email]
        )
    except Exception as exc:
        raise self.retry(exc=exc, countdown=60)  # retry after 1 min on failure

9 Common Pitfalls & How to Avoid Them

⚠ Pitfall 1: Race Conditions on Stock Decrement

Two users buy the last item simultaneously. Both see stock=1, both succeed, you oversell. This is a real problem with MongoDB.

✓ Fix

Use MongoDB's atomic findOneAndUpdate with a filter that includes stock >= quantity. If it returns null, stock was insufficient. Never read stock then write separately.

result = Product.objects(id=pid, "variants__variant_id"=vid, "variants__stock__gte"=qty).update_one(
    dec__variants__S__stock=qty  # atomic decrement
)
if not result: raise OutOfStockError()
⚠ Pitfall 2: Storing Product Prices in Cart Only

If a product's price changes after someone adds it to cart, their cart now shows the wrong price. At checkout, which price do you charge?

✓ Fix

Store price_at_add in the cart item when it's added. On checkout, use this stored price. Also snapshot product name, image, and variant details into the Order document — so historical orders are never affected by product edits.

⚠ Pitfall 3: Processing Webhooks Twice

Stripe can occasionally send the same webhook event more than once (network retries). If you mark an order "paid" twice, you send two confirmation emails and may double-decrement stock.

✓ Fix

Make your webhook handler idempotent. Before processing, check if the Stripe event ID has already been recorded. If yes, return 200 immediately without processing.

if Payment.objects(stripe_event_id=event.id).first():
    return HttpResponse(status=200)  # already processed
⚠ Pitfall 4: Fetching Data in useEffect Everywhere

New React developers write useEffect(() => { fetch(...) }, []) everywhere. This leads to loading state bugs, race conditions on fast navigation, and no caching — every page visit refetches everything.

✓ Fix

Use TanStack Query (React Query) for all server data. It handles caching, deduplication, background refetching, and loading/error states consistently. One library replaces dozens of useEffect patterns.

⚠ Pitfall 5: Not Versioning Your API from Day One

You build the API without versioning. Six months later you need to change a response shape — but your mobile app (or third-party integrations) break. Now you're stuck.

✓ Fix

Always prefix all endpoints with /api/v1/ from the very first endpoint. It costs nothing to add now and saves you later. When you release breaking changes, add /api/v2/ and keep v1 alive temporarily.

⚠ Pitfall 6: Treating MongoDB Like a Relational Database

Writing lots of multi-collection queries because you normalized everything. MongoDB's strength is embedding related data. If you always need product + variants + category together, embed variants in the product document.

✓ Fix

Rule of thumb: Embed data that is always read together and doesn't grow unboundedly (variants, addresses, order items). Reference by ObjectId when data is shared across many documents (products in orders — but snapshot name/price at order time).

⚠ Pitfall 7: Building the Admin Dashboard First

Many beginners build the admin panel first because it seems manageable. You spend 4 weeks on it before having a working storefront. You also won't know which admin features matter until you have real store data.

✓ Fix

Follow the roadmap: auth → products → cart → checkout → emails → THEN admin. Use Django Admin (the built-in one) as a temporary admin interface during development. Build your custom admin dashboard last, when you know exactly what you need.

⚠ Pitfall 8: Skipping Tests Until the End

"I'll write tests later" means no tests ever. You also lose confidence in refactoring, which leads to a codebase full of "don't touch this — it works and I don't know why."

✓ Fix

Write at minimum: API integration tests (use DRF's APITestCase) for every endpoint as you build it. Test happy paths + key failure cases (out of stock, invalid coupon, unauthorized access). For React, test business logic in custom hooks with Vitest. Use Postman collections as a quick sanity check after each deploy.
